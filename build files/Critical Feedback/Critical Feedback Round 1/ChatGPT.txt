# OpenRA Knowledge Base Manual Review

# Overall Verdict

This is an unusually ambitious and valuable OpenRA reference. It succeeds far better as a **source-grounded architectural encyclopedia** than as a traditional beginner's manual. The strongest aspects are its breadth, consistent chapter template, code-path tracing, dependency explanations, extension-point coverage, and explicit focus on determinism. The weakest aspects are educational progression, practical onboarding, uneven depth distribution, and the lack of end-to-end workflows that connect concepts into real development tasks.

In its current form, I would describe it as:

> Excellent reference documentation, good maintainer documentation, and only moderate newcomer documentation.

The manual clearly targets new contributors, modders, and maintainers, but it serves the latter two audiences much more effectively than the first.

---

# Priority Issues (Highest Impact First)

## 1. The manual is reference-first, not learning-first

### Problem
Most chapters begin with architecture details, interfaces, caches, lifecycle ordering, and internal implementation details before readers have built a mental model.

### Where
Entire manual, especially Parts 1–4.

### Why It Matters
New contributors will understand *what exists* but not *why it exists*.

### Suggested Fix
Add a recurring section near the start of every chapter:

- What problem does this subsystem solve?
- What would break if it didn't exist?
- What does a modder need to know?
- What does an engine contributor need to know?

---

## 2. Missing End-to-End Walkthroughs

### Problem
Systems are explained individually, but there are few complete journeys.

### Missing Examples

- Click unit → order issued → activity queued → movement → rendering update
- Weapon fired → projectile → warhead → damage → death
- YAML trait → TraitInfo → runtime trait → actor lifecycle
- AI decision → order → lockstep simulation

### Why It Matters
Readers learn systems in isolation.

### Suggested Fix
Add 5–10 "vertical slice" walkthroughs that cross multiple chapters.

---

## 3. Too Much Implementation Detail Before Conceptual Grounding

### Problem
Readers encounter advanced constructs such as:

- TraitDictionary
- TypeDictionary
- ISingleInstanceInit
- IRulesetLoaded
- Interface caches

before they fully understand the actor model.

### Where
ECS chapter, Activities chapter, RMG chapters.

### Why It Matters
Creates unnecessary cognitive overload.

### Suggested Fix
Move advanced implementation details into dedicated sections:

- Advanced Notes
- Internal Implementation
- Maintainer Notes

---

## 4. No Contributor Onboarding Roadmap

### Problem
Learning paths are topic-oriented rather than task-oriented.

### Missing Examples

- First engine PR
- First custom trait
- First custom weapon
- First custom AI module
- First renderer modification

### Suggested Fix
Add:

# Part 0.5 — Contributor Roadmaps

---

## 5. Practical Modding Examples Are Too Sparse

### Problem
Architecture is explained well, but practical implementation examples are limited.

### Why It Matters
Modders need examples more than architecture.

### Suggested Fix

Add recipe-style guides:

- Add a new weapon
- Add a status effect
- Add a building
- Add a support power
- Add a new actor trait

Include before/after YAML examples.

---

## 6. Coverage Is Heavily Skewed Toward RMG

### Problem
Random Map Generation receives substantially more depth than many core systems.

### Why It Matters
Documentation weight does not match likely reader demand.

### Suggested Fix

Either:

- Expand Rendering, UI, Lua, Networking, and Official Mods

or

- Condense RMG sections

## 11. Little Discussion of Testing Strategies

### Problem
Contributors need guidance on:

- Unit tests
- Regression tests
- Sync testing
- Replay testing

### Suggested Fix
Add a dedicated Testing chapter.

---

## 12. Few Troubleshooting Workflows

### Problem
Readers learn architecture but not debugging methodology.

### Suggested Fix

Add troubleshooting playbooks:

- Unit won't move
- Weapon doesn't fire
- Trait isn't loading
- Desync investigation

---

## 13. File Path Accuracy May Drift Rapidly

### Problem
The manual is tightly coupled to specific source file locations.

### Why It Matters
Future OpenRA versions may invalidate references.

### Suggested Fix
Emphasize concepts more than exact file locations and line references.

---

## 14. Some Chapters Read Like Source-Code Commentary

### Problem
Several sections resemble annotated source listings.

### Why It Matters
Readers may lose sight of design intent.

### Suggested Fix
Compress implementation detail and elevate architectural rationale.

---

## 15. Missing "What Should I Read Next?" Sections

### Problem
Learning flow often stops at chapter boundaries.

### Suggested Fix
End chapters with:

- What you've learned
- What to read next
- Related systems

# Final Score

**Overall Rating: 4.1 / 5**

The manual is already close to being an exceptional OpenRA reference. The next major improvement is not adding more subsystem coverage—it is improving **learning flow, onboarding, task-oriented guidance, and practical workflows** so that newcomers can grow into the architectural material rather than being dropped directly into it.

Current state:

> Excellent reference documentation  
> Good maintainer documentation  
> Moderate newcomer documentation

Target state:

> Excellent reference documentation  
> Excellent maintainer documentation  
> Excellent contributor onboarding resource  
> Excellent practical modding guide

The highest-leverage improvements are:

1. Contributor roadmaps
2. End-to-end workflows
3. Practical modding tutorials
4. Better onboarding structure
5. Expanded Rendering/UI/Networking coverage
6. Reduced implementation-detail overload
7. More troubleshooting and debugging guidance
